\chapter{A Case Study: NYC Crashes}


Every day, New Yorkers travel across the city in countless ways:
walking to school, biking to work, driving across town, or riding
buses and subways. Most of these journeys end safely, but some result
in crashes that cause injuries, delays, and sometimes fatalities. Can
data help us understand when crashes occur, where they are most
common, and what factors are associated with them? In this chapter, we
investigate these questions using a real dataset from NYC Open
Data. Along the way, we will apply many of the ideas developed
throughout this book, including importing data, cleaning messy
records, exploring patterns with tables and graphs, working with dates
and times, handling missing values, and interpreting results
carefully. This chapter brings together many of the concepts and
skills we have learned and shows how statistical thinking and data
science can be used together to answer questions with real-world data.

Along the way, we will also ask a few inferential questions. Are severe
crashes more common on business days than on weekends? What is a
reasonable estimate of the overall severe-crash rate for the week, and
how uncertain is that estimate? Which recorded features are associated
with crash severity? These questions are natural candidates for
count-based hypothesis testing, estimation, and regression. They do not
call for a t-test, because the response is binary rather than normally
distributed.


\section{Importing and Inspecting the Data}

We begin by reading the comma-separated values file, or CSV file. A CSV
file is a plain text file that stores a data table. Each row is one
record, and commas separate the values in that row. The raw data was
downloaded from NYC Open Data Portal and included all the crashes
reported in the week containing the Labor Day of 2025. The first row
of the CSV file contains the variable names.

\showChunk{R}{Code/nyc_crashes.R}{read}

The object \texttt{df} is now a data frame. A data frame is R's usual
way to store a rectangular data table. Before doing any analysis, we
check its structure, number of rows, number of columns, and first few
variable names.

\showChunk{R}{Code/nyc_crashes.R}{inspect}

This quick check tells us what kind of object we have. It also helps us
catch problems early. For example, if the row count were zero, we would
know the file did not load correctly.

The original variable names use capital letters and spaces. Those names
are readable, but they are awkward to type in R. We first create a new
set of names by changing letters to lower case and replacing
periods with underscores.

\showChunk{R}{Code/nyc_crashes.R}{rename_build}

Now we apply the new names to the data frame and inspect the result.

\showChunk{R}{Code/nyc_crashes.R}{rename_apply}

This step does not change the data values. It only changes the labels on
the columns. A name such as \code{NUMBER.OF.PERSONS.INJURED} becomes
\code{number\_of\_persons\_injured}, which is easier to type and less
likely to cause mistakes.

Next we convert the crash date from text into a date object. This lets R
compare dates in a sensible way.

\showChunk{R}{Code/nyc_crashes.R}{dates_convert}

Now we check the first date, the last date, and the number of crashes on
each date.

\showChunk{R}{Code/nyc_crashes.R}{dates_check}

This is a small step, but it is one of the most useful habits in data
analysis: check whether the data match the story you were told. The raw
file spans eight dates. We want the Sunday--Saturday week around Labor
Day, so we define that target window.

\showChunk{R}{Code/nyc_crashes.R}{filter_week}

The logical variable \texttt{in\_week} is \texttt{TRUE} for rows inside
the target week and \texttt{FALSE} for rows outside it. We count how
many rows pass that test, keep only those rows, and then check the date
range again.

\showChunk{R}{Code/nyc_crashes.R}{filter_apply}

Check-in: Why do we check the date range both before and after
filtering? What mistake might this catch?

The raw file has a \texttt{location} column that stores coordinates as
text. It also has separate numeric columns for latitude and longitude.
We look at the three columns together.

\showChunk{R}{Code/nyc_crashes.R}{location_preview}

If \texttt{location} repeats the same information as latitude and
longitude, it may not be useful for our analysis. We also count how many
different location strings appear and how many are missing or blank.

\showChunk{R}{Code/nyc_crashes.R}{location_unique}

\section{Missing Values and Data Quality}

Missing values are not just empty spaces. They are clues about how the
data were collected. A missing borough or zip code might mean the crash
happened on a highway, at an unusual address, or in a record where the
location was not entered completely.

We start with the borough column. The function \texttt{trimws()} removes
extra spaces at the beginning and end of each entry. Then we convert
blank borough names into \texttt{NA}, R's missing-value marker.

\showChunk{R}{Code/nyc_crashes.R}{borough_clean}

Zip codes can also have impossible values. If the zip code column is
numeric, a zero should not be treated as a real New York City zip code,
so we recode it as missing.

\showChunk{R}{Code/nyc_crashes.R}{zip_clean}

Now we count missing boroughs, missing zip codes, and rows where both
are missing.

\showChunk{R}{Code/nyc_crashes.R}{missing_counts}

These counts do not tell us why the values are missing. They tell us
where to look more carefully.

Location coordinates need their own check. New York City latitudes are
around 40 degrees north, and longitudes are around 74 degrees west.
Coordinates such as \texttt{(0,0)} are not in New York City. We mark
coordinates as bad if they are zero-like or outside a broad box around
the city.

\showChunk{R}{Code/nyc_crashes.R}{geo_bad}

After counting the bad coordinates, we replace them with \texttt{NA}.
Then we count rows where either latitude or longitude is missing.

\showChunk{R}{Code/nyc_crashes.R}{geo_recode}

Sometimes a row has latitude and longitude but no zip code. In that
case, we might be able to fill in the zip code later by using a map. We
create \texttt{fillable\_zip} for rows with missing zip codes and
present coordinates.

\showChunk{R}{Code/nyc_crashes.R}{fillable_make}

To compare days of the week, we create a weekday variable. The
\texttt{levels} argument puts the days in calendar order instead of
alphabetical order.

\showChunk{R}{Code/nyc_crashes.R}{weekday}

Now we count fillable zip codes by weekday and calculate the proportion
for each day.

\showChunk{R}{Code/nyc_crashes.R}{fillable_table}

Check-in: If one day has a higher proportion of fillable zip codes, does
that prove the data collectors worked differently on that day? Why or
why not?

\subsection{A few inferential questions}

The descriptive tables above invite a few simple inferential checks.
Because severity is a yes-or-no outcome, the appropriate comparison is
not a t-test. Instead, we can compare the 2-by-2 table of severe and
non-severe crashes on business days and weekends with Pearson's
chi-squared test of independence. That test works with counts, which is
the right scale for this data.

\runR{Code/nyc_crashes.R}{nyc-severity-inference}

% Show the 2x2 table and chi-squared test as a small, focused step.
\showChunk{R}{Code/nyc_crashes.R}{severity_inference_table}
\showChunk{R}{Code/nyc_crashes.R}{severity_chisq}

The contingency table is
\inlnR{```tab_business_severe```}[nyc-sev-tab][vbox]
and the chi-squared test gives a p-value of
\inlnR{```round(chisq_business_severe$p.value, 4)```}[nyc-sev-p].
For this week, the evidence is not strong enough to say that severe
crashes differ by day type.

We can also estimate the overall severe-crash rate for the week. The
point estimate is \inlnR{```round(severe_rate, 3)```}[nyc-sev-rate],
and the exact 95\% binomial confidence interval is
\inlnR{```round(severe_rate_ci$conf.int[1], 3)```}[nyc-sev-lo]
to \inlnR{```round(severe_rate_ci$conf.int[2], 3)```}[nyc-sev-hi].
This interval is a direct estimate for a binary response; it does not
rely on a normal approximation for the raw crash outcomes.

To make the binomial calculation concrete, here are the raw counts of
severe and non-severe crashes used by the binomial test:

\showChunk{R}{Code/nyc_crashes.R}{severity_binom_table}

Interpretation: the point estimate above reports the fraction of crashes
in this sampled week that were classified as severe. The exact binomial
confidence interval gives a range of plausible values for that
underlying proportion. A narrow interval means the estimate is relatively
stable (we observed many crashes); a wide interval means more sampling
uncertainty (fewer crashes). These values describe the observed week and
should not be interpreted as causal effects.

\section{Summary}

This case study used the same habits we have practiced throughout the
book. We loaded a dataset, checked its shape, cleaned names, converted
dates, filtered rows, looked for missing values, created new variables,
and compared groups with tables and proportions.

The biggest lesson is not a single fact about New York City traffic. The
lesson is the workflow. Real datasets need careful questions at every
step: Did the file load? Are the dates right? Are missing values coded
consistently? Do the variables mean what their names suggest? Are we
describing a pattern, or are we claiming a cause?

Open data makes these questions available to anyone with patience and a
computer. City agencies, federal agencies, scientific groups, sports
leagues, and many other organizations share data that students can
explore. NYC Open Data is a good place to look for more examples. In
addition to crash records, it includes widely used datasets such as 311
Service Requests, which record calls and online reports about noise,
heat, street conditions, trash, and many other city services. With the
tools from this book, you can begin with one clear question, check the
data carefully, and build an honest answer one step at a time.
